第3章 单一职责原则 ​

原书位置：《大话设计模式【Java溢彩加强版】》第 3 章（趣味篇名常见作「电子阅读器 vs 手机」/「阅读器 vs 手机」一类表述）
类别：设计原则（SOLID 之 S；不是 GoF 23 模式之一）
痛点关键词：一个类身兼多职、改 A 伤到 B、测试回归面失控
一句话意图：按「变更理由」给类划界，让一类变更尽量只落在一处。

书里一句话常写成：

就一个类而言，应该仅有一个引起它变化的原因。（Single Responsibility Principle，缩写 SRP。）

它不是说「一个类只能有一个方法」，而是：几条彼此独立的变更理由被写进同一个类里时，这个类就管得太宽了。

一、30 秒卡片（一眼回忆用） ​

  • 它解决什么变化：同一文件里混了几条独立变更线（例如计分怎么算、分数怎么展示），改其中一条就被迫动到另一条。
  • 抓住三个词就够：
    • 职责：更常说成「一类变更理由」——谁改、为什么改，最好集中在一处。
    • 杂糅：几种变更理由堆在一个类里，git blame 永远是一锅粥。
    • 拆分后的麻烦：类多了，边界、传参、联调就得说清楚，这就是拆的代价。
  • 容易跟什么搞混：为了读起来顺，把一个大方法拆成小方法——那是同一个变更理由里整理结构；SRP 管的是不同变更理由该不该同居一室。

一个月后只看本节就能回忆起这章 80% 内容。

二、不用原则约束时怎么写（先看烂代码） ​

需求（玩具）：方块游戏需要「根据消除行数与当前关卡算分，并在控制台打印分数面板」。

v1：把规则与展示焊在一起

java
// 玩具：分数面板 —— 计分规则与控制台输出写在同一方法的同一层级
public class ScorePanelV1 {
    public void refreshAndPrint(int lines, int level) {
        int score = lines * 10 + level * 5;
        System.out.println("=== SCORE ===");
        System.out.println("Lines: " + lines + " Level: " + level);
        System.out.println("Points: " + score);
    }
}

v1 的痛点：

  • 两类变更理由耦合：计分公式调整 vs 控制台/本地化/换 Skin 展示方式，却落在同一类同一方法里，改展示常要摸到业务公式。
  • 测试与演进：想单独单测「计分」或单独替换「输出通道」，被迫启动整段流程或造 I/O。
  • 可读性的假象：代码很短，但责任域并不单一——这是 SRP 讨论里最常见的「短类依然违规」。

三、一步步演化 ​

3.1 v2：同一类内先分离「算分」与「打印」 ​

（下面这版是练习里写过的一版：先把公式和 println 分开，读代码会顺很多。）

java
public class ScorePanelV2 {
    public void refreshAndPrint(int lines, int level) {
        int score = getScore(lines, level);
        printScore(lines, level, score);
    }

    public int getScore(int lines, int level) {
        return lines * 10 + level * 5;
    }

    public void printScore(int lines, int level, int score) {
        System.out.println("=== SCORE ===");
        System.out.println("Lines: " + lines + " Level: " + level);
        System.out.println("Points: " + score);
    }
}

好在哪 / 还差什么：

  • 好：阅读路径清晰——公式与打印分离，refreshAndPrint 只做编排；小体量下维护更顺手。
  • 差：真项目里若是两个人、两种发版节奏分别改「规则」和「界面」，它们仍共用一个 ScorePanel，类这一层还是捆在一起。

3.2 v3：把两条变更理由拆到不同类型（示意终局之一） ​

java
/** 只关心：给定局面怎么记分——规则变更主要落在这里 */
public final class ScoreRules {
    public int scoreFor(int lines, int level) {
        return lines * 10 + level * 5;
    }
}

/** 只关心：怎么把已有结果展示出来——换行、换 UI、换日志，主要落在这里 */
public final class ConsoleScoreView {
    public void render(int lines, int level, int score) {
        System.out.println("=== SCORE ===");
        System.out.println("Lines: " + lines + " Level: " + level);
        System.out.println("Points: " + score);
    }
}

/** 只做拼积木：先算分，再交给视图去画 */
public class ScorePresentation {
    private final ScoreRules rules = new ScoreRules();
    private final ConsoleScoreView view = new ConsoleScoreView();

    public void refreshAndPrint(int lines, int level) {
        int score = rules.scoreFor(lines, level);
        view.render(lines, level, score);
    }
}

好在哪 / 还差什么：

  • 好：计分规则与展示各自的修改者更能「各改各文件」，测试可以分别覆盖纯规则与 I/O（后者可用替身替换 ConsoleScoreView）。
  • 差：文件变多；render 要不要认识 lines/level、视图模型长什么样，后面都要另作约定。

四、书上类比落在代码里是什么 ​

「阅读器 vs 手机」讲的是一回事：功能可以很多，但代码里别把事情绑成一种改法。 手机本身多用途不是问题；问题是一种类因为好几条互不搭界的变更理由老被一起打开、一起改。

随手留个对照：

观念在代码里长什么样
「引起变化的原因」需求迭代、线上事故、性能专项、换了第三方 SDK、换了 UI 规范……凡是迫使打开这个类去改的动机。
职责过宽同一类里 if 分支的一条支路因业务 A 改、另一条支路因业务 B 改，却共享字段与生命周期。
拆分收益变更局部化、测试聚焦、代码审查范围缩小。
拆分代价类型增多、跳转增多、组件间契约必须写清；过度拆会出现无意义的薄包装。

五、反例与代价 ​

  • 什么时候不该「为了 SRP 而拆」：
    • 代码量极小、变更理由在未来可见周期内高度一致（例如只在同一需求里一起改的脚本）；
    • 机械地把每个方法变成一个类，但抽象轴只有一个、也不会扩展——只剩导航噪音。
  • 代价：
    • 类/文件数量上升，新人找入口的时间变长（需要好的分层与命名）；
    • 调试链路变长：从「哪种子类在跑」（多态）到「跨组件契约是否一致」，都要花时间；
    • 联调成本：界面与领域逻辑分离后，必须通过显式接口/DTO/事件传递信息，沟通与回归测试范围要重新划分。
  • 误用样子：两个类总是一起发布、一起回滚、共享同一生命周期，却被拆成「貌似 SRP」的两份——结果只是把一个合理内聚单元硬撕开，测试与事务边界都变痛苦。

六、回到自己的代码 ​

  • 练习里的 v2：先把算分和打印拆开，日常维护已经好不少。
  • 什么时候再动 v3：规则和外观经常各改各的、各上各的线，却仍然共用一个类，就可以考虑把类也撕开。
  • 土办法：改 UI 和改业务很少落在同一条工单，却老是动到同一份文件——先写下来「这份文件到底会因为哪几件完全不同的事被打开」，再决定要不要拆。

七、自检三问（合上书自答） ​

先不看下文，用一句话或几个关键词写下你的答案；想完再展开对照。

  1. 这一章里，「阅读器 / 手机」的类比想让你区分的是哪类混在一起会一起变坏的东西？所谓稳定的一侧通常指什么？
  2. 「职责」更接近业务功能分工，还是更接近独立变更理由的一条责任线？
  3. 拆分之后的收益与代价各列举你最在意的两点。
答完后再展开：参考表述
  1. 搅在一起会变糟的：几条互不相干的变更理由塞在同个类里，改一条牵连另一条，回归面变大。所谓「相对稳定」多半是你想守住的那条对外边界（接口、领域规则怎么对外承诺），各项目叫法不同，心里有谱即可。
  2. 职责：SRP 里更像「这条变更线的专责」，而不是把菜单栏功能数一遍。多问自己：打开这个类，通常是因为外面哪几类完全不同的需求？ 若答案很多且彼此独立，就该琢磨拆。
  3. 收益：改动范围缩小，评审和测试好对焦；界面和规则各改各的时候，不至于互相踩脚。
    代价：类多了要跳转；边界上要有契约、DTO、联调；拆错了就是两套维护成本。

正文开头的定义句与这里的「变更理由」说的同一件事，一句短、一句长而已。

八、一句话口诀 ​

按变更理由划界拆类，牵一发动一片时先怀疑职责搅在一起。
