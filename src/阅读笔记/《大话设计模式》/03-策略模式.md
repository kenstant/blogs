# 第2章 策略模式

> **类别**：行为型  
> **痛点关键词**：收银/促销规则变多、`switch` 与业务流程绑死、扩展一种活动要改多处  
> **一句话意图**：**把一族可互换的算法封装成策略，让使用方依赖抽象而非具体分支。**

## 一、30 秒卡片（一眼回忆用）

- **它解决什么变化**：**计费/促销怎么算**（原价、折扣、满减等）经常变、还会新增；**收款流程**（读单价、数量、汇总）相对稳定。
- **关键角色**：
  - **`Strategy`（抽象策略）**：声明「给定单价、数量，怎么算出实付」；书中常用抽象类 `CashSuper`；  
  - **`ConcreteStrategy`**：`CashNormal`、`CashRebate`、`CashReturn` 等具体算法；  
  - **`Context`**：`CashContext` 持有当前策略，对外提供统一接口（如 `getResult`），**把「选哪一种策略」集中在一个地方**；  
  - **客户端**：只和 `Context` 或抽象策略打交道，避免长篇 `switch` 散落。
- **它和谁容易混**：与上一章**简单工厂**——两边都可能出现「一堆算法子类 + 一个根据类型 `new` 的分支」。**区别在意图与收口**：简单工厂侧重**创建**；策略侧重**运行时算法的封装与替换**，常由 `Context` 持有策略并在业务中委托。

> 一个月后只看本节就能回忆起这章 80% 内容。

## 二、不用模式时怎么写（先看烂代码）

**需求**：做一个商场收银软件，根据商品的单价和数量收费；后续要支持单独打折、全场折扣、满 300 减 100 等。

**v1 直觉写法**（逻辑堆在 `main` + `switch`）：

```java
package design_pattern.chap2.v1;

import java.util.Scanner;

public class CashV1 {
    public static void main(String[] args) {
        double price = 0d;      //商品单价
        int num = 0;            //商品购买数量
        double totalPrices = 0d;//当前商品合计费用
        double total = 0d;      //所有商品总计费用
        int discount = 0;       //商品销售模式 1.原价 2.八折 3.七折

        Scanner sc = new Scanner(System.in);
        do {
            System.out.println("请输入商品销售模式 1.原价 2.八折 3.七折: ");
            discount = Integer.parseInt(sc.nextLine());
            System.out.println("请输入商品单价: ");
            price = Double.parseDouble(sc.nextLine());
            System.out.println("请输入商品数量: ");
            num = Integer.parseInt(sc.nextLine());
            System.out.println();

            if (price > 0 && num > 0) {
                switch (discount) {
                    case 1:
                        totalPrices = price * num;
                        break;
                    case 2:
                        totalPrices = price * num * 0.8;
                        break;
                    case 3:
                        totalPrices = price * num * 0.7;
                        break;
                }
                total = total + totalPrices;
                System.out.println();
                System.out.println("单价: " + price + "元,数量: " + num + ",合计: " + totalPrices + "元.");
                System.out.println();
                System.out.println("总计: " + total + "元.");
                System.out.println();
            }
        } while (price > 0 && num > 0);
    }
}
```

**v1 的痛点**：

- 每加一种促销就要改 `switch`，**容易牵动整个输入—输出流程**。  
- 「算法」和「控制台交互 / 累计总额」搅在一起，**难复用、难单测**。  
- 书中会继续加「满减」等分支，**继续堆 `case` 只会更痛**。

## 三、一步步演化

### 3.1 v2：先拆算法 + 简单工厂创建

先把「怎么算单笔实付」拆到 `CashSuper` 子类里，再用 **`CashFactory`** 根据类型 `new` 出具体策略；`main` 里只保留 `acceptCash`。

```java
package design_pattern.chap2.v2;

import java.util.Scanner;

public class CashV2 {

    static abstract class CashSuper {
        abstract double acceptCash(double price, int num);
    }

    static class CashNormal extends CashSuper {
        @Override
        public double acceptCash(double price, int num) {
            return price * num;
        }
    }

    static class CashRebate extends CashSuper {
        private final double rebate;

        public CashRebate(double rebate) {
            this.rebate = rebate;
        }

        @Override
        public double acceptCash(double price, int num) {
            return price * num * rebate;
        }
    }

    static class CashReturn extends CashSuper {
        private final double condition;
        private final double returnPrice;

        public CashReturn(double condition, double returnPrice) {
            this.condition = condition;
            this.returnPrice = returnPrice;
        }

        @Override
        double acceptCash(double price, int num) {
            double result = price * num;
            if (condition > 0 && result > condition) {
                result = result - Math.floor(result / condition) * returnPrice;
            }
            return result;
        }
    }

    static class CashFactory {
        public static CashSuper createCashSuper(int cashType) {
            switch (cashType) {
                case 1:
                    return new CashNormal();
                case 2:
                    return new CashRebate(0.8d);
                case 3:
                    return new CashRebate(0.7d);
                case 4:
                    return new CashReturn(300d, 100d);
            }
            return new CashNormal();
        }
    }

    public static void main(String[] args) {
        double price;      //商品单价
        int num;            //商品购买数量
        double totalPrices;//当前商品合计费用
        double total = 0d;      //所有商品总计费用
        int discountType;       //商品销售模式 1.原价 2.八折 3.七折

        Scanner sc = new Scanner(System.in);
        do {
            System.out.println("请输入商品销售模式 1.原价 2.八折 3.七折: ");
            discountType = Integer.parseInt(sc.nextLine());
            System.out.println("请输入商品单价: ");
            price = Double.parseDouble(sc.nextLine());
            System.out.println("请输入商品数量: ");
            num = Integer.parseInt(sc.nextLine());
            System.out.println();

            if (price > 0 && num > 0) {
                CashSuper cashSuper = CashFactory.createCashSuper(discountType);
                totalPrices = cashSuper.acceptCash(price, num);
                total += totalPrices;
                System.out.println();
                System.out.println("单价: " + price + "元,数量: " + num + ",合计: " + totalPrices + "元.");
                System.out.println();
                System.out.println("总计: " + total + "元.");
                System.out.println();
            }
        } while (price > 0 && num > 0);
    }

}
```

**好在哪 / 还差什么**：

- **好**：每种促销一段独立类，`acceptCash` 各司其职；**新增一种算法主要是加子类**（工厂里仍要补 `case`）。算法与流程有一定分离。  
- **差**：客户端仍直接感知 **`CashFactory` + `CashSuper` 两个名字**；书中会继续收口——用 **`Context`** 包住「当前用哪种策略」，调用方只面对一个入口（见 v3）。

### 3.2 v2 → v3：引入 Context（策略模式的关键一步）

在算法子类不变的前提下，增加 **`CashContext`**：构造时根据类型选好具体 `CashSuper`，对外只暴露 `getResult`。客户端从：

`CashSuper c = CashFactory.createCashSuper(type); c.acceptCash(...)`

变为：

`new CashContext(type).getResult(...)`。

**新增/变化的重心**（完整类见下一节「终版」）：

```java
    static class CashContext {
        private final CashSuper cashSuper;

        public CashContext(int cashType) {
            switch (cashType) {
                case 2:
                    cashSuper = new CashRebate(0.8d);
                    break;
                case 3:
                    cashSuper = new CashRebate(0.7d);
                    break;
                case 4:
                    cashSuper = new CashReturn(300d, 100d);
                    break;
                default:
                    cashSuper = new CashNormal();
            }
        }

        public double getResult(double price, int num) {
            return this.cashSuper.acceptCash(price, num);
        }
    }
```

**好在哪**：业务流程侧**只认 `Context`**，与「今天到底打几折」的具体类名解耦一层；也便于以后把 Context 做成**可注入策略**（与书后文、练习的延伸一致）。**仍要记住**：若继续用 `switch` 装配策略，**类型枚举一变仍要改 Context**——更彻底的做法可结合配置、Map、工厂等（本书循序渐进）。

## 四、模式登场（终版）

**v3 完整代码**（与本地练习工程一致）：

```java
package design_pattern.chap2.v3;

import java.util.Scanner;

public class CashV3 {

    static abstract class CashSuper {
        abstract double acceptCash(double price, int num);
    }

    static class CashNormal extends CashSuper {
        @Override
        public double acceptCash(double price, int num) {
            return price * num;
        }
    }

    static class CashRebate extends CashSuper {
        private final double rebate;

        public CashRebate(double rebate) {
            this.rebate = rebate;
        }

        @Override
        public double acceptCash(double price, int num) {
            return price * num * rebate;
        }
    }

    static class CashReturn extends CashSuper {
        private final double condition;
        private final double returnPrice;

        public CashReturn(double condition, double returnPrice) {
            this.condition = condition;
            this.returnPrice = returnPrice;
        }

        @Override
        double acceptCash(double price, int num) {
            double result = price * num;
            if (condition > 0 && result > condition) {
                result = result - Math.floor(result / condition) * returnPrice;
            }
            return result;
        }
    }

    static class CashContext {
        private final CashSuper cashSuper;

        public CashContext(int cashType) {
            switch (cashType) {
                case 2:
                    cashSuper = new CashRebate(0.8d);
                    break;
                case 3:
                    cashSuper = new CashRebate(0.7d);
                    break;
                case 4:
                    cashSuper = new CashReturn(300d, 100d);
                    break;
                default:
                    cashSuper = new CashNormal();
            }
        }

        public double getResult(double price, int num) {
            return this.cashSuper.acceptCash(price, num);
        }
    }

    public static void main(String[] args) {
        double price;      //商品单价
        int num;            //商品购买数量
        double totalPrices;//当前商品合计费用
        double total = 0d;      //所有商品总计费用
        int discountType;       //商品销售模式 1.原价 2.八折 3.七折

        Scanner sc = new Scanner(System.in);
        do {
            System.out.println("请输入商品销售模式 1.原价 2.八折 3.七折: ");
            discountType = Integer.parseInt(sc.nextLine());
            System.out.println("请输入商品单价: ");
            price = Double.parseDouble(sc.nextLine());
            System.out.println("请输入商品数量: ");
            num = Integer.parseInt(sc.nextLine());
            System.out.println();

            if (price > 0 && num > 0) {

                CashContext cashContext = new CashContext(discountType);
                totalPrices = cashContext.getResult(price, num);
                total += totalPrices;
                System.out.println();
                System.out.println("单价: " + price + "元,数量: " + num + ",合计: " + totalPrices + "元.");
                System.out.println();
                System.out.println("总计: " + total + "元.");
                System.out.println();
            }
        } while (price > 0 && num > 0);
    }
}
```

**UML / 角色对应**

| 模式角色 | 上例中的类 |
| --- | --- |
| Strategy | `CashSuper`（抽象策略） |
| ConcreteStrategy | `CashNormal`、`CashRebate`、`CashReturn` |
| Context | `CashContext` |
| Client | `main`（仅依赖 `CashContext` 的用法） |

## 五、反例与代价

- **什么时候不该用**：促销规则**长期固定且只有一种写法**；或为两三个 `if` 就上全套抽象 + `Context`，**阅读成本高于收益**。策略适合「算法族明显会长的变化轴」。  
- **代价**：  
  - **类/内部类变多**，调试要在「Context ↔ 具体策略」之间跳；  
  - 若装配策略仍写死 **`switch`**，**新增策略要改 Context（或工厂）**——只是把 `switch`**挪了位置**，并未自动消失；  
  - 策略接口设计过宽时，**子类被迫共享不需要的参数**，会出现「空实现」坏味道。  
- **误用样子**：每个 `if` 都机械地拆成一个类，但**没有稳定抽象**、子类之间无法替换；或 `Context` 里堆满**与算法无关的业务**（上帝类），**名存实亡的策略**。

## 六、写法说明：单文件嵌套 vs 书中分文件

本节 **二～四** 里的 `CashV1` / `CashV2` / `CashV3` 已是按书上演化顺序整理的**练习代码**：用**一个 `.java`、若干 `static` 内部类**收在一处，方便在笔记里通读和对照版本差异。

书中示意图往往是**每个具体策略、Context、工厂各自独立成文件**，只是排版与粒度不同；**类职责、协作关系与 v1→v2→v3 的推进**与多文件写法一致。若拆文件，通常会将 `CashSuper` 与各 `ConcreteStrategy`、`CashContext` 提到顶层类，包路径仍可与 `design_pattern.chap2.v*` 一类命名对齐。

**后续可改进（与布局无关）**：控制台提示可写明「4. 满 300 减 100」；策略装配若不想改 `switch`，可再试注入、注册表或配置驱动。

## 八、一句话口诀

> **封装会变的算法成策略，用 Context 收口调用，让分支退出业务流程中心。**
