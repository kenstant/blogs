# 第4章 开放-封闭原则（OCP）

> **原书位置**：《大话设计模式【Java溢彩加强版】》**第 4 章**（趣味篇名「考研求职两不误」）  
> **类别**：**设计原则**（SOLID 之 **O**；不是 GoF 23 模式之一）  
> **痛点关键词**：每加一类需求就回头改一大片、`if`/分支链疯长、老代码经不起频繁改动  
> **一句话意图**：**对扩展开放**：新行为主要靠**加代码**接住；**对修改封闭**：已稳定的抽象与高频复用实现尽量不因每一种新变种而被反复撕开。

书里常见表述：**软件实体应当可以扩展，但不可修改**（开放-封闭原则，**Open-Closed Principle**，缩写 **OCP**）。实战中更务实的读法是：**把「会经常变的那一维」抽到扩展点一侧**，而让**核心业务或对外契约**一侧尽量不因每一种新特例而大改——「完全不修改」多半是理想界限，本章重在**把修改面压到小、可到局部**。

## 一、30 秒卡片（一眼回忆用）

- **它解决什么变化**：同一业务（如成绩总评）下，**算法的种类或分支**在变；希望**相对稳定**的多半是：**抽象契约（接口层 / 规范的对外形状）**，以及尽量不反复被打开的那批主干实现的组织方式——具体实现可按新类型按需增加。  
- **关键角色**：  
  - **抽象侧**：定下「要完成什么事」的稳定接口或抽象骨架；  
  - **扩展侧**：针对不同考试类型的**具体算法/策略**，通过多态接上抽象；  
  - **组装/选型处**：决定在运行时选哪一个具体实现——这里仍可能有一处分支，但往往可收敛到一处注册或工厂（书中亦常配合简单工厂、策略等你已见过的形态）。  
- **它和谁容易混**：和第 3 章 **SRP** 都谈改动，但 SRP 更问「**谁在同一个类里承担了过多变更理由**」；OCP 更问「**新需求来时，你能不能主要通过扩展接住，而不是没完没了改旧的稳定核心**」。也常和「**策略模式**」（第 2 章）在代码长得像——区别是：**策略是解决一类问题的模式手段**；**OCP 是说何时值得那样拆的规则**。

> 一个月后只看本节就能回忆起这章 80% 内容。

## 二、不用原则约束时怎么写（先看烂代码）

**需求（玩具）**：成绩单根据考试类型汇总一门「总评」，考研与求职的计分口径不同；未来还可能加国考、保研认定等。

```java
enum ExamKind { MASTER, JOB }

class ScoreReportServiceV1 {
    double overall(ExamKind kind, Map<String, Double> grades) {
        if (kind == ExamKind.MASTER) {
            return grades.get("written") * 0.35 + grades.get("interview") * 0.65;
        }
        if (kind == ExamKind.JOB) {
            return grades.get("written") * 0.40
                    + grades.get("tech") * 0.40
                    + grades.get("hr") * 0.20;
        }
        throw new IllegalArgumentException("unknown kind");
        // 每加一个考试类型都要回来改这个大方法
    }
}
```

**v1 的痛点**：

- **修改面随行就市地膨胀**：加一个 `ExamKind` 就要动同一方法里的分支，易出现**遗漏、判断错误**、多人协作合并冲突。  
- **可读性与演进**：主干逻辑与各色特例交织，**审稿与测试**难以只针对单一考试类型收口。  
- **与需求的直觉错位**：书里「两不误」一类的故事想强调：**新增一条赛道**本应像「加一个插件」，而不是每次都把旧赛道代码再摊开改一遍。

## 三、一步步演化

### 3.1 v2：把算法拆成小类

很自然的第一步：**每种考试类型单独一块**。

```java
class MasterExamScore {
    double overall(Map<String, Double> grades) {
        return grades.get("written") * 0.35 + grades.get("interview") * 0.65;
    }
}

class JobExamScore {
    double overall(Map<String, Double> grades) {
        return grades.get("written") * 0.40
                + grades.get("tech") * 0.40
                + grades.get("hr") * 0.20;
    }
}

// 加国考 / 保研认定 → 再 new 一个同类的小类……
```

**好在哪 / 还差什么**：

- **好**：公式互不污染，可读单元更清晰。  
- **差**：调用方若没有统一抽象，**「选 Master 还是 Job」**仍可能在别处长成 `switch`——还没把「选型」收口。

### 3.2 v3：抽象 + 注册表（终局示意）

```java
interface OverallPolicy {
    double overall(Map<String, Double> grades);
}

class MasterOverallPolicy implements OverallPolicy {
    public double overall(Map<String, Double> grades) {
        return grades.get("written") * 0.35 + grades.get("interview") * 0.65;
    }
}

class JobOverallPolicy implements OverallPolicy {
    public double overall(Map<String, Double> grades) {
        return grades.get("written") * 0.40 + grades.get("tech") * 0.40 + grades.get("hr") * 0.20;
    }
}

class ScoreReportFacade {
    private final Map<ExamKind, OverallPolicy> registry = new HashMap<>();

    ScoreReportFacade() {
        registry.put(ExamKind.MASTER, new MasterOverallPolicy());
        registry.put(ExamKind.JOB, new JobOverallPolicy());
    }

    double overall(ExamKind kind, Map<String, Double> grades) {
        return registry.get(kind).overall(grades);
    }
    // 加国考：new 一个 OverallPolicy，再 registry.put 一行
}
```

**好在哪 / 还差什么**：

- **好**：每种 `OverallPolicy` 不因新考试题型而再被撕开；接线集中在 **注册表**，比在方法里散落 `if` 好审得多。  
- **差**：类与跳转多了；注册表配错要到运行期才露头。

## 四、原则和代码怎么对齐

本章不是再教一个新「模式学名」，而是在问：**哪一种变化轴值得被抬成扩展点**。落实在代码上的常见组合拳包括：

| 书里想守住的 | 在工程里可能的落点 |
| --- | --- |
| **对扩展开放** | 新考试类型 ≈ 新的 `OverallPolicy` 实现，或新的策略类族 |
| **对修改封闭** | 尽量不反复编辑已上线的**稳定算法主体**与各类型彼此独立的单元测试 |
| **依赖抽象** | 调用方只看到 `OverallPolicy`，而不是散落在各处的具体公式分支 |

和第 2 章 **策略**的关系：本章是在说**该不该**为了一类可变行为引入「可插拔的一组算法」；策略模式常常正是实现 OCP 的一种手段，但二者层级不同——**别把「接口 + 一堆实现」自动等同于「遵守了 OCP」**，还要看**修改是否还是被牵得到处跑**。

## 五、反例与代价

- **什么时候不该硬上这一套**：  
  - **分支极少且短期内稳定**（一两种写法、`if`-`else` 一次写完几年不动）——强加接口层多半是 **YAGNI**，维护成本先于收益到账。  
  - **抽象的契约自己都还在剧烈变动**：若顶层接口隔三差五随产品推翻，空谈「封闭」没有意义，应先让需求与领域模型**稳定一版**。  
- **代价**：  
  - **类和文件变多**，阅读路径从「一页看完」变成「在多类型间跳转」。  
  - **调试**：打断点、看日志时往往是接口类型，追到**真正干活的类**要多一步。  
  - **选型/注册错了**更隐蔽：注册表、工厂、依赖注入配置一旦配错，编译期抓不住。  
- **误用样子**：**给每个 Service 都无脑套一层接口**，而业务上几乎**只有一种实现**，也看不到第二条实现时间表——这层接口并没有换来「扩展时少改既有代码」，只增加了**跳转与样板**。

```java
// 形似分层、难为扩展：NationwideExamService 长期只有一个实现类
interface NationwideExamService { void submit(); }
class NationwideExamServiceImpl implements NationwideExamService {
    public void submit() { /* ... */ }
}
```

## 六、回到自己的代码

- **从 v2 到 v3**：拆出 `MasterExamScore` / `JobExamScore` 仍可能在外层长满 `switch`；**OverallPolicy + 注册表**才把「选哪一种计算器」收口到一处。  
- **自查**：最近一次加需求，你是主要以「**新开一个类 / 新方法**」完成，还是以「**在三个老方法各插一段**」完成？后者的次数若长期偏高，可读本章与第 2 章对照复盘。  
- **团队里的真实情况**：不少人见过**给几乎每个业务 Service 都套一层同名接口**，而库里长期只有一种实现的情形；可以挑一两个模块问自己：**删掉接口之后，有哪类未来改动会从「局部加实现」沦陷成「大改一大片」？** 若答案想不出来，那层拆分就更像习惯而非 OCP 收益。

3. 和第 3 章 **SRP**：一个管「拆分变更理由」，一个管「扩展方式」——举一个**两句话能说明白两者差别**的日常例子。

1. **开放去接纳的**：会变的那一维——例如「考试类型对应的计分算法」越来越多。  
   **尽量少动的一侧**：对外稳定的契约与主干组织方式——例如 `OverallPolicy` 这类抽象、`ScoreReportFacade` 的调用入口；已上线的各类型算法实现本身，理想情况下是**加新类**而不是回去改旧公式。  
2. **算偏离**：若每加一种变体仍要在三五个 `switch` / `if` 里各补一段，修改面并没被压小，只是类变多了。  
   **收敛手段**：注册表集中 `put`、简单工厂/策略里的 **Context**、配置或插件清单驱动、依赖注入按类型绑定——共性都是**把「选哪一个实现」收到一处**。  
3. **例子（成绩单玩具）**：  
   - **SRP**：计分规则与控制台展示写在同一个类里——改打印格式不该逼你打开计分公式，所以要按**变更理由**拆类。  
   - **OCP**：新增「国考」计分——理想是 **new 一个 `OverallPolicy` + 注册表加一行**，而不是回到 v1 那个大 `if` 里把考研、求职的代码再摊开改一遍。

## 八、一句话口诀

> **依赖抽象接住变体，主干少动靠加扩展点。**
